## Why This Matters

The existing public reference points for AI experimentation skills all fail one of three tests:

| Reference | Problem |
|---|---|
| GrowthBook / Statsig / Eppo docs | Platform-bound. Excellent on their own platform, useless if you switch vendors. |
| Optimizely / LaunchDarkly playbooks | Vendor-locked + sales-flavoured. Heavy on flag management, light on validity. |
| Kohavi *Trustworthy Online Controlled Experiments* | Definitive but book-form — not operational at the agent prompt level. |
| Generic "A/B test" skills floating around | Treat A/B as the only method; no holdout, no switchback, no quasi-experimental, no MAB. Treat results as binary win/lose; no Decision Class, no SRM gate, no pre-committed decision rule. |

The actual problem of running rigorous experiments lives at four levels:

| Level | Failure mode | Where this suite solves it |
|---|---|---|
| **Lifecycle** | Jumping straight from idea to launch with no spec | `experimentation` orchestrator with hard transitions backlog → spec → runbook → readout |
| **Prioritisation** | ICE as a vibe; no feasibility gate; underpowered tests get launched | `experiment-backlog` with binary-blocking feasibility gate (traffic, metric latency, method fit, population stability) |
| **Method choice** | "Just A/B test it" regardless of interference / cluster effects / metric latency | `experiment-spec` + `method-selector.md` — picks the right method for the surface |
| **Readout** | Peeking; calling underpowered directional tests "wins"; ignoring SRM | `experiment-readout` with SRM as a blocking gate and Decision Class enforcement |

Each level cannot be carried in a single ≤200-line skill — hence the family of 5.

## Key Design Decisions

### Lifecycle-decomposed, NOT method-decomposed
The natural-feeling decomposition is one skill per method (`ab-test`, `holdout`, `switchback`, `mab`). That's wrong. Method is one *choice inside the spec*, not a top-level skill. The right decomposition is the lifecycle: idea → backlog → spec → runbook → readout. A user planning a switchback test still needs the same spec rigor and the same readout discipline as one running a vanilla A/B. Method lives in `experiment-spec/references/method-selector.md`.

### Decision Class on every artefact — Causal / Directional / Instrumentation
Every spec, runbook, and readout carries an explicit Decision Class label:

- **Causal** — adequately powered, properly randomised, validity-clean. Allowed to use words like *significant*, *winner*, *lift*, *ship*.
- **Directional** — underpowered or quasi-experimental. Forbidden from using *significant* / *winner* / *lift* / *ship*. Can only inform direction.
- **Instrumentation** — pre-launch QA test (e.g. exposure event firing correctly). No business decision allowed.

This kills the most common experimentation failure mode: directional tests being interpreted as causal wins because nothing on the report told the reader the difference.

### SRM (Sample Ratio Mismatch) as a blocking hard gate
Before any metric is read out, `experiment-readout` runs a chi-squared SRM check. If the assignment ratio differs significantly from the planned ratio (e.g. 50/50 actually splitting 47/53), readout is BLOCKED and the experiment is logged as `SRM-FAIL`. Most public A/B tooling treats SRM as a soft warning. It is not — SRM means the assignment system is broken and metric differences cannot be attributed to the variant.

### Platform-agnostic spec; PostHog-primary runbook
`experiment-spec` is fully platform-neutral — it produces the same artefact whether the team uses PostHog, GrowthBook, Statsig, LaunchDarkly, Optimizely, or Eppo. Only `experiment-runbook` binds to a platform. PostHog gets a full first-class binding (flags, exposure events, dashboards, ramp plan, rollback). The other 5 vendors get a single mapping table that the user adapts. This avoids carrying 6 deep platform integrations across the suite — only the runbook layer cares about the platform.

### Append-only cumulative learnings log
`experiment-readout` always appends to `docs/experiments/learnings.md`, regardless of decision (SHIP / ITERATE / KILL / INCONCLUSIVE / SRM-FAIL). A team that runs 30 tests and writes nothing learns nothing. A team that appends one paragraph per test accumulates institutional memory worth more than any single result.

### Pre-committed decision rule
Every spec must declare its decision rule before launch — e.g. "ship if primary lift CI lower bound > 0 AND no guardrail breach." This is checked against in `experiment-readout`. Post-hoc rule changes are explicitly forbidden by `decision-class-rules.md`. This is the single biggest defence against motivated reasoning at readout time.

### Peek policy declared upfront
Every spec declares the peek policy: fixed-horizon (no peek), sequential testing (e.g. mSPRT, always-valid p-values), or Bayesian decision rule. Mixing peek policies between spec and readout invalidates the test. `experiment-readout` flags any test whose actual peek pattern differs from the declared policy and downgrades the Decision Class to Directional.

### Living portfolio with explicit feasibility gate
`experiment-backlog` is not just an ICE-sorted list. Every candidate must clear a binary-blocking 4-part gate (traffic, metric latency, method fit, population stability) before entering the active list. High-ICE candidates that fail feasibility get parked in a `Rejected` table with the reason recorded, so they can be re-evaluated when feasibility improves. This stops the "high impact / unrunnable" backlog rot.

## How to Use

| Trigger phrase | What runs |
|---|---|
| "design an experiment" / "A/B test this" | Full orchestrator (`experimentation`) — routes through whichever lifecycle stage you're in |
| "what should we test next" | `experiment-backlog` direct |
| "spec this experiment" / "design this A/B test" | `experiment-spec` direct |
| "set up the experiment" / "wire this up in PostHog" | `experiment-runbook` direct |
| "read out this experiment" / "did the test win" / "ship or kill" | `experiment-readout` direct |

Output structure:
```
docs/experiments/
  backlog.md          <- experiment-backlog (living portfolio)
  learnings.md        <- experiment-readout (append-only cumulative log)
  specs/              <- experiment-spec
    YYYY-MM-DD-<name>-spec.md
  runbooks/           <- experiment-runbook
    YYYY-MM-DD-<name>-runbook.md
  analyses/           <- experiment-readout
    YYYY-MM-DD-<name>-analysis.md
```

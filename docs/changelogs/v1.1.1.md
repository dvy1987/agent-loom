# v1.1.1 — AlphaEval Research Integration & Post-Application Hardening

**Date:** 2026-04-20

This release ingests findings from **"AlphaEval: Evaluating Agents in Production"** (Lu, Xu, Zhang et al., 2026 — arXiv preprint, credibility 8/12) — a benchmark of 94 real-world tasks sourced from 7 companies, evaluated across 4 commercial scaffolds (Claude Code, Codex, GitHub Copilot, Cursor). Eight empirically-grounded insights were applied to 6 skills. Additionally, the entire learn-from skill family was upgraded with a mandatory Post-Application Hardening Cycle, bringing it to parity with improve-skills for security and compression enforcement.

---

## Breaking Changes

- None.

## What Was Learned (AlphaEval Paper)

The paper exposed five structural blind spots in the skill library:

1. **Scaffold choice is a first-order performance driver** — the same model (Opus 4.6) varied by 11 points across scaffolds (64.41 on Claude Code vs 53.45 on Codex). The `agent-builder` skill designed agent architectures without considering scaffold performance characteristics.

2. **High aggregate scores mask low economic value** — a model scoring 48/100 overall delivered more economic value than one scoring 62/100 because it won on high-value domains. The `eval-rubric-design` skill had no concept of value-weighting dimensions by business impact.

3. **Cascade dependency is the #1 pipeline failure mode** — early-stage errors invalidate all downstream steps. The `eval-pipeline` skill evaluated end-to-end but lacked per-step intermediate validation for multi-step agent pipelines.

4. **Agents produce cross-section logical inconsistencies** — contradictory statements within long-form deliverables (e.g., different market sizes on different pages). Neither `eval-judge` nor `eval-rubric-design` had internal consistency as a standard evaluation dimension.

5. **Agents optimize explicit goals while violating implicit domain constraints** — a top failure mode even with good task decomposition. The `process-decomposer` had no guidance to probe for unstated professional norms.

The paper also validated two existing approaches: format adherence as a hard gate (already in `eval-rubric-design`) and multi-paradigm evaluation composition (already in `eval-pipeline`'s 3-layer stack). These were marked KEEP CURRENT.

## What Changed (AlphaEval Insights → 8 Skill Improvements)

- **`eval-rubric-design` (v1.0→v1.1)** — added "Internal consistency" to the dimension library for long-form outputs (checks for contradictions across sections). Added gotcha on value-weighting dimensions by business impact — high aggregate scores can mask low performance on the dimensions that matter most.

- **`eval-judge` (v1.0→v1.1)** — added gotcha for long-form internal consistency checks: scan for numeric contradictions, conflicting claims, and logical inconsistencies across sections of any output >1 page.

- **`eval-pipeline` (v1.0→v1.1)** — added gotcha on per-step intermediate validation for multi-step agent pipelines. Cascade dependency is the documented #1 failure mode — checkpoints between stages, not just end-to-end evaluation.

- **`agent-builder` (v1.0→v1.1)** — added two gotchas: (1) scaffold choice as a first-order performance driver with specific performance data, (2) cross-agent validation checkpoints to prevent synergy blindness (26% cost overruns documented when agents optimized components independently).

- **`process-decomposer` (v1.0→v1.1)** — added gotcha about implicit domain constraints. Agents miss unstated professional norms, regulatory expectations, and domain conventions. Step 0 should now probe: "Are there industry norms, compliance rules, or unwritten expectations that an outsider wouldn't know?"

- **`reality-check` (v1.0→v1.1)** — added gotcha with production benchmark ceiling: best agent configs achieve <65% on production tasks. "Production-ready" and "human-level" agent claims should be scored with extreme skepticism. Includes IR failure rate breakdown (30% hallucinations, 35% imprecise retrieval, 10% positive-info bias).

## What Changed (Post-Application Hardening Cycle)

The learn-from family previously mentioned "200-line gate" and "validate-skills" as bullet points but lacked an enforced sequence. Meanwhile, improve-skills had a rigorous per-skill cycle with explicit security scanning, compression, and validation. This gap meant insights could be applied to skills without verifying the result was secure, correctly sized, or structurally sound.

- **`learn-from` orchestrator (v2.0→v2.1)** — replaced vague "Post-Application Rules" with an explicit **Post-Application Hardening Cycle** — a 4-step mandatory sequence run on every modified/created skill after changes are applied:
  1. Modified-skill security sweep (all `secure-*` skills on the resulting skill — separate from source ingestion scan)
  2. Version bump + citation
  3. 200-line gate via `compress-skill` / `split-skill`
  4. `validate-skills` ≥10/14 (runs after any compress/split so the final form is validated)

  Updated call graph to show the two-pass security model: source scan → apply → modified-skill scan → compress/split → validate. Updated impact report format to surface post-apply results per skill.

- **`learn-from-paper` (v2.0→v2.1)** — aligned Step 6 (Apply) to explicitly reference the mandatory Post-Application Hardening Cycle.

- **`learn-from-article` (v2.0→v2.1)** — aligned Step 5 (Match and Apply) to explicitly reference the hardening cycle.

- **`learn-from-repo` (v2.0→v2.1)** — aligned Step 5 (Match and Apply) to explicitly reference the hardening cycle.

- **`learn-from-chat` (v1.0→v1.1)** — expanded Step 5 (Apply) from 6 to 8 sub-steps, adding explicit modified-skill security sweep (#6) and 200-line gate (#7), with validate-skills as the final gate (#8). This sub-skill got the most detailed treatment because it owns its own apply sequence rather than deferring to the orchestrator.

## Expected Outcome & Impact

### For the eval skills (eval-rubric-design, eval-judge, eval-pipeline)
**Outcome:** Evaluations of agent outputs will now catch three documented failure modes that were previously invisible: value-score disconnect, long-form inconsistencies, and cascade dependency in multi-step pipelines.
**Impact:** More reliable eval results. Fewer "the eval passed but the output was wrong" situations. Rubrics for agent outputs will naturally include internal consistency and value-weighting when applicable, catching the kinds of failures that AlphaEval documented across 94 real production tasks.

### For agent-builder
**Outcome:** Agent architecture decisions will now account for scaffold performance characteristics and cross-agent optimization effects.
**Impact:** Better scaffold/platform selection during architecture design. Reduced risk of 26% cost overruns from synergy blindness. Agents designed with cross-boundary validation checkpoints instead of assuming independent optimization is sufficient.

### For process-decomposer
**Outcome:** Task decomposition will probe for implicit domain constraints that agents consistently miss.
**Impact:** Fewer failures from agents optimizing for stated goals while violating unstated professional norms — a failure mode that persists even with good decomposition, per AlphaEval's findings across HR, finance, procurement, and healthcare domains.

### For reality-check
**Outcome:** Claims about agent capabilities will be scored against empirical production benchmarks.
**Impact:** More grounded assessments. "Production-ready" claims now face a documented ceiling of 64.41/100 from the best configuration tested on authentic business tasks. Prevents over-optimistic evaluation of agent products.

### For the learn-from family (hardening cycle)
**Outcome:** Every skill modification through learn-from now goes through the same rigor as improve-skills: security scan of the result, size enforcement, and validation.
**Impact:** Closes the gap where insights could be applied and leave skills in an insecure, oversized, or structurally degraded state. The two-pass security model (scan source, then scan modified skill) prevents both poisoned inputs and accidentally introducing security issues during the edit.
